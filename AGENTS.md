# CTX — AI Development Pipeline

CTX is a unified pipeline for managing development tasks across multiple AI providers.
It orchestrates a full cycle: DETECT, CONTEXT, TASK, BRAINSTORM, PLAN, EXECUTE, DONE.

## Universal Support

CTX now works on **all providers** with **identical functionality**: Claude Code, Codex CLI, Gemini CLI, OpenCode

✨ **One command `/ctx` everywhere — same functionality!**

| Provider  | Mode       | Skill File                  | Setup Guide                    |
|-----------|------------|----------------------------|--------------------------------|
| Claude Code | MCP native | `skills/ctx-universal-full/` | Works out of the box (.mcp.json) |
| Codex CLI  | CLI wrapper | `skills/ctx-universal-full/` | `node scripts/ctx-setup.js codex` |
| Gemini CLI | CLI wrapper | `skills/ctx-universal-full/` | `node scripts/ctx-setup.js gemini` |
| OpenCode   | CLI wrapper | `skills/ctx-universal-full/` | `node scripts/ctx-setup.js opencode` |

**Quick setup for all:**
```bash
node scripts/ctx-setup.js all
```

The universal skill automatically detects the mode (MCP or CLI wrapper) and provides the same API across all providers.

## Pipeline State (Shared across all providers)

All providers work with the same state files:

- **Pipeline state:** `.data/pipeline.json` — read/write
- **Action log:** `.data/log.jsonl` — append-only, one JSON object per line
- **Session logs:** `.sessions/YYYY-MM-DD-HHmm.md`
- **Project index:** `.data/index.json` — generated by indexer

**Important:** You can switch between providers — state is preserved!

### Access Methods

| Provider    | Access Method                      | Tool/Script          |
|-------------|------------------------------------|----------------------|
| Claude Code | MCP tools (native)                 | `ctx_get_pipeline`, `ctx_set_stage`, etc. |
| Codex CLI   | MCP tools (if available), else CLI wrapper (Node.js) | `ctx_*` tools or `scripts/ctx-cli.js` |
| Gemini CLI  | CLI wrapper via bash               | `scripts/ctx-cli.js` |
| OpenCode    | CLI wrapper via Bash tool          | `scripts/ctx-cli.js` |

### CLI Wrapper API (for non-MCP providers)

```bash
# Get pipeline state
node scripts/ctx-cli.js get_pipeline

# Set stage
node scripts/ctx-cli.js set_stage --stage context

# Update pipeline fields
node scripts/ctx-cli.js update_pipeline --patch '{"lead":"gemini","task":"Build API"}'

# Log action
node scripts/ctx-cli.js log_action --entry '{"action":"code_written","file":"src/api.js"}'

# Log error
node scripts/ctx-cli.js log_error --entry '{"error":"TypeError","solution":"Add null check"}'
```

Codex note: if the host exposes the `ctx-hub` MCP server, use MCP tools first and keep CLI/file modes only as fallback.

See `CTX_UNIVERSAL.md` for detailed installation and usage.

## Agents

Base agents are defined in `agents/*.md`. Each agent has a role and a pipeline stage:

| Agent            | Role                              | Stage     |
|------------------|-----------------------------------|-----------|
| `architect`      | Decomposition, API contracts      | PLAN      |
| `implementer`    | Writing code per approved plan    | EXECUTE   |
| `reviewer`       | Code review, quality checks       | EXECUTE   |
| `tester`         | Tests, coverage                   | EXECUTE   |
| `researcher`     | Research, PoC, exploration        | BRAINSTORM|
| `documenter`     | README, API docs, changelogs      | EXECUTE   |
| `pipeline-controller` | Pipeline orchestration       | internal  |
| `session-logger` | Session save to GitHub Issues     | DONE      |

To use an agent: read its markdown file from `agents/<name>.md` and follow its instructions.

## Pipeline stages

```
DETECT -> CONTEXT -> TASK -> BRAINSTORM -> PLAN -> EXECUTE -> DONE
```

Each stage reads and writes `.data/pipeline.json`. The file has this structure:

```json
{
  "stage": "detect",
  "lead": "codex",
  "task": null,
  "context": {},
  "brainstorm": {},
  "plan": {},
  "log": []
}
```

## Rules

1. All state lives in `.data/pipeline.json` — always read before modify, then write back
2. Append logs to `.data/log.jsonl` — never overwrite, only append
3. Agents are independent markdown files — read them when needed
4. Consilium providers must not see each other's answers until synthesis
5. Lessons go to GitHub Issues with `lesson` label
6. `.sessions/` is in `.gitignore` — session logs are private
