# CTX ā€” AI Development Pipeline

CTX is a unified pipeline for managing development tasks across multiple AI providers.
It orchestrates a full cycle: DETECT, CONTEXT, TASK, BRAINSTORM, PLAN, EXECUTE, DONE.

## How it works in Codex CLI

Codex does NOT support MCP. All pipeline state is stored as files:

- **Pipeline state:** `.data/pipeline.json` ā€” read/write via Read/Write tools
- **Action log:** `.data/log.jsonl` ā€” append-only, one JSON object per line
- **Session logs:** `.sessions/YYYY-MM-DD-HHmm.md`
- **Project index:** `.data/index.json` ā€” generated by indexer

### File operations instead of MCP tools

| MCP tool (not available)      | Codex equivalent                                          |
|-------------------------------|-----------------------------------------------------------|
| `ctx_get_pipeline()`          | Read `.data/pipeline.json`                                |
| `ctx_set_stage(stage, data)`  | Read pipeline.json, set `stage` field + merge data, Write |
| `ctx_update_pipeline(patch)`  | Read pipeline.json, merge patch object, Write             |
| `ctx_log_action(action)`      | Append JSON line to `.data/log.jsonl`                     |
| `ctx_list_agents()`           | List filenames in `agents/` directory                     |
| `ctx_log_error(err)`          | Append JSON line to `.data/log.jsonl` with type: "error"  |

## Agents

Base agents are defined in `agents/*.md`. Each agent has a role and a pipeline stage:

| Agent            | Role                              | Stage     |
|------------------|-----------------------------------|-----------|
| `architect`      | Decomposition, API contracts      | PLAN      |
| `implementer`    | Writing code per approved plan    | EXECUTE   |
| `reviewer`       | Code review, quality checks       | EXECUTE   |
| `tester`         | Tests, coverage                   | EXECUTE   |
| `researcher`     | Research, PoC, exploration        | BRAINSTORM|
| `documenter`     | README, API docs, changelogs      | EXECUTE   |
| `pipeline-controller` | Pipeline orchestration       | internal  |
| `session-logger` | Session save to GitHub Issues     | DONE      |

To use an agent: read its markdown file from `agents/<name>.md` and follow its instructions.

## Pipeline stages

```
DETECT -> CONTEXT -> TASK -> BRAINSTORM -> PLAN -> EXECUTE -> DONE
```

Each stage reads and writes `.data/pipeline.json`. The file has this structure:

```json
{
  "stage": "detect",
  "lead": "codex",
  "task": null,
  "context": {},
  "brainstorm": {},
  "plan": {},
  "log": []
}
```

## Rules

1. All state lives in `.data/pipeline.json` ā€” always read before modify, then write back
2. Append logs to `.data/log.jsonl` ā€” never overwrite, only append
3. Agents are independent markdown files ā€” read them when needed
4. Consilium providers must not see each other's answers until synthesis
5. Lessons go to GitHub Issues with `lesson` label
6. `.sessions/` is in `.gitignore` ā€” session logs are private
