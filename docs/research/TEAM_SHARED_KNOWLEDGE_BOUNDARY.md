# Shared Knowledge Boundary and Access Model

This brief defines the first-pass boundary between the current single-user CTX runtime and any future shared/team knowledge layer.

## Why this exists

Epic `#396` starts only after the single-user runtime and product surfaces are stable. The goal here is to define the future shared scope without pushing enterprise or multi-tenant requirements back into the current execution plan.

## Boundary map

### Single-user runtime memory

The current CTX runtime remains the canonical owner of:

- `.data/` pipeline state and operator runtime controls
- local session logs, local KB entries, and decision exports
- user-scoped provider setup, failover, and outage recovery state

This layer is intentionally private-by-default. It can export artifacts, but it does not become shared workspace state automatically.

### Shared team knowledge workspace

A future shared layer may contain:

- approved shared knowledge entries
- reusable templates promoted from local artifacts
- governed decision archives visible to a team workspace

This layer must sit above the local runtime. Promotion into shared state is explicit and reviewable.

### Enterprise and deferred scope

The following stay outside the current build commitment:

- tenant isolation and compliance controls
- marketplace/community distribution ideas
- legal/governance requirements that change the Phase 0-2 runtime

These remain research inputs, not runtime requirements.

## Access model

The first-pass access model is intentionally small:

- `workspace_owner`
  - manages members and workspace policy
  - approves shared exports/templates when policy requires it
- `contributor`
  - writes shared knowledge and publishes templates within policy
  - does not manage membership or policy
- `viewer`
  - reads approved shared artifacts
  - does not mutate shared state

Important constraint: no role gets implicit access to another user`s private runtime state, provider credentials, or local recovery controls.

## Governance assumptions

- Shared knowledge is promoted from local artifacts through explicit publish flows.
- Local provider credentials and private runtime state never become shared workspace defaults.
- Review applies to shared exports/templates, not to a user`s private session loop.

## Non-goals

- replacing the current single-user runtime with a multi-user state machine
- auto-syncing every local KB write into a shared database
- sharing provider API keys or local model configs across users
- committing to marketplace, tenant, or compliance work in the current roadmap

## Guardrails for the roadmap

- Do not add team-write semantics to the current local KB or pipeline state machine.
- Do not require enterprise tenancy design to finish current roadmap work.
- Keep shared/team knowledge as a separate research stream until its publish/review boundary is explicit.
