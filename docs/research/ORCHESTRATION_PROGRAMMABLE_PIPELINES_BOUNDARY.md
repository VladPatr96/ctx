# Orchestration Boundary and Programmable Pipeline Model

This brief defines how CTX should treat advanced orchestration ideas such as decomposition, branching, batch execution, and programmable pipeline builders.

## Why this exists

Epic `#397` is intentionally a research stream. The goal is to map these ideas into one place without turning them into an implied runtime commitment before the execution model is fully stable.

## Concern map

### Task decomposition

Decomposition stays a planning concern first.

- The runtime may record parent-child intent and artifact references.
- The runtime should not autonomously spawn recursive child pipelines yet.
- Future planners or operator flows can promote decomposition into an explicit action later.

### Branching and parallel paths

Branching is a governed operator action, not a default execution rule.

- Runtime contracts may carry branch labels or ancestry metadata.
- Only one active path should remain the normal runtime assumption for now.
- Future operator surfaces can compare alternate branches once recovery and merge policies are explicit.

### Batch execution

Batch execution belongs to a later scheduling layer.

- The current runtime should stay centered on one tracked task.
- Queue-friendly task envelopes are acceptable as metadata.
- Future batch orchestration can build on normalized status, telemetry, and recovery hooks.

### Programmable pipeline builder

A builder or DSL is explicitly deferred.

- The contract layer must stabilize before any public programmable surface exists.
- Reusable pipeline templates can be researched now, but not productized as a build commitment.
- Any future builder must sit on top of validated step/task contracts and explicit recovery rules.

## Core runtime boundary

The current runtime is responsible for:

- task and step state machines
- deterministic execution policies and artifact manifests
- provider lifecycle hooks, recovery, and traceability
- metadata that can later support decomposition, branching, or batch scheduling

The runtime is not responsible for:

- autonomous recursive decomposition
- parallel branch orchestration by default
- queue schedulers or batch dispatch engines
- public DSL or visual builder promises

## Deferred product surfaces

The following remain deferred until runtime stabilization is proven:

- visual branching and replay tools
- queue and batch scheduling surfaces
- programmable pipeline templates and DSL work
- visual pipeline builder experiences

## Rollout guards

- Do not add automatic recursive task spawning to the current runtime.
- Do not make branching or batch execution default behavior before explicit policy exists.
- Do not commit to a public DSL or visual builder before runtime contracts are stable.
- Keep advanced orchestration in research until a later ADR promotes a specific slice.

## Non-goals

- promising advanced orchestration as part of the current roadmap commitment
- treating batch mode as a trivial extension of the interactive task loop
- freezing a public orchestration DSL before contract stability exists
- moving product-surface ambition ahead of operator control and auditability
