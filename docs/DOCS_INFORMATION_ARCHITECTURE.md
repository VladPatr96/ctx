# CTX Documentation Information Architecture

This document defines the target documentation layout for CTX and records how the current repo documentation should migrate into that structure.

## Goals

- keep one canonical product landing surface
- separate user-facing setup/reference docs from contributor/process docs
- treat `skills/*/SKILL.md` and `agents/*.md` as source material, not the final documentation surface
- remove root-level markdown drift over time by migrating or archiving it

## Canonical Docs Surfaces

| Target surface | Purpose | Current sources |
|---|---|---|
| `docs/index` | Product overview, install entrypoint, feature map | `README.md` |
| `docs/setup/providers` | Provider setup and migration between Claude/Codex/Gemini/OpenCode flows | `CTX_UNIVERSAL.md`, `OPENCODE_AUTO_SETUP.md`, provider setup notes |
| `docs/reference/interfaces` | Canonical reference for CLI commands and MCP tools | `docs/reference/CLI_MCP_REFERENCE.md`, interface manifest |
| `docs/reference/dashboard` | Canonical reference for dashboard HTTP endpoints and Electron desktop shell contracts | `docs/reference/dashboard/DASHBOARD_DESKTOP_REFERENCE.md`, dashboard desktop manifest |
| `docs/reference/project-memory` | Exported session history and durable decision snapshots | `.sessions/*.md`, `docs/ADR_*.md`, export generators |
| `docs/reference/skills` | Reference for CTX skills and generated skill docs | `skills/*/SKILL.md`, `CTX_SKILLS_*`, skill automation notes |
| `docs/reference/agents` | Reference for agent roles and runtime behavior | `agents/*.md`, `AGENTS.md` |
| `docs/architecture/adr` | Durable architecture decisions | `docs/ADR_*` |
| `docs/research/*` | Research briefs for deferred or post-stabilization streams | team knowledge boundary, advanced orchestration boundary, future enterprise follow-ups |
| `docs/contributing/project-memory` | Project workflow, issue/process memory model | `docs/GITHUB_PROJECTS_MEMORY_WORKFLOW.md`, `WORKFLOW.md` |
| `docs/contributing/testing` | Test migration policy, E2E verification, reliability practices | `docs/TEST_MIGRATION_CLASSIFICATION.md`, test result notes |
| `docs/release/versioning` | Release discipline, versioning, packaging rules | `docs/release/VERSIONING.md`, publish workflow metadata |
| `docs/migration/*` | Competitive/legacy migration guides | planned follow-up work in epic `#390` |
| `docs/archive/*` | Legacy root docs that remain useful but are no longer canonical | old plans, status notes, one-off reports |

## Migration Rules

### Canonical now

- `README.md`
- `docs/ADR_PHASE0_SQLITE_FIRST.md`
- `docs/GITHUB_PROJECTS_MEMORY_WORKFLOW.md`
- `docs/TEST_MIGRATION_CLASSIFICATION.md`
- `docs/release/VERSIONING.md`
- `docs/reference/CLI_MCP_REFERENCE.md`
- `docs/reference/dashboard/DASHBOARD_DESKTOP_REFERENCE.md`
- `docs/reference/project-memory/SESSION_DECISION_EXPORTS.md`
- `docs/setup/providers/PROVIDER_MIGRATION_GUIDE.md`

### Migrate into canonical docs

- root setup docs such as `CTX_UNIVERSAL.md` and `OPENCODE_AUTO_SETUP.md`
- workflow/process notes such as `WORKFLOW.md`
- testing result notes and provider testing markdown currently living at repo root

### Keep as source material

- `skills/*/SKILL.md`
- `agents/*.md`
- `AGENTS.md`

These files are operational/runtime inputs first. They should feed generated or curated reference docs instead of being treated as the final product docs surface.
