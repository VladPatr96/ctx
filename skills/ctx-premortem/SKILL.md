# /ctx-premortem — Risk Analysis (Tigers Framework)

Структурированный pre-mortem анализ для любого плана, PRD или фичи.

## Использование

```
/ctx-premortem <описание продукта/фичи/плана>
/ctx-premortem --file <путь к PRD.md>
/ctx-premortem <тема> --timeline 30d
```

### Флаги
- `--file <path>` — прочитать PRD/план из файла
- `--timeline <N>d` — горизонт запуска (по умолчанию 14 дней)
- `--save` — сохранить результат в файл `PreMortem-[name]-[date].md`

## Фреймворк: Tigers / Paper Tigers / Elephants

### Tigers (Реальные угрозы)
Проблемы, основанные на фактах, опыте или очевидной логике. Требуют действий.

Классификация по срочности:
- **Launch-Blocking** — должны быть решены ДО запуска
- **Fast-Follow** — решить в течение 30 дней после запуска
- **Track** — мониторить, решать если станет проблемой

### Paper Tigers (Раздутые страхи)
Выглядят страшно, но маловероятны или переоценены. Документируй для alignment, но не трать ресурсы.

### Elephants (Слон в комнате)
О чём команда не говорит, но стоило бы. Неозвученные допущения, которые никто не валидирует. Требуют расследования.

## Workflow

### Шаг 1: GATHER

1. Прочитай PRD/план (из аргумента или `--file`)
2. Определи: продукт, целевой рынок, ключевые допущения, таймлайн
3. Если доступен контекст проекта — запусти `node "${CLAUDE_PLUGIN_ROOT}/scripts/ctx-indexer.js"` для карты проекта

### Шаг 2: IMAGINE FAILURE

Представь: продукт запущен через [timeline] дней. Он ПРОВАЛИЛСЯ.
- Клиенты не приняли продукт
- Метрики не достигнуты
- Репутация пострадала

Что пошло не так? Что мы пропустили? В чём были слишком уверены?

### Шаг 3: CLASSIFY

Для каждого потенциального провала определи тип:

**Tigers** — спроси себя:
- Есть ли факты/данные, подтверждающие этот риск?
- Видел ли я подобный провал раньше?
- Логика очевидна или это домысел?

**Paper Tigers** — спроси себя:
- Почему этот страх раздут?
- Какова реальная вероятность?
- Что изменится если мы его проигнорируем?

**Elephants** — спроси себя:
- Обсуждала ли команда это открыто?
- Какие допущения стоят за этим молчанием?
- Что будет если это окажется правдой?

### Шаг 4: ACTION PLANS

Для каждого Launch-Blocking Tiger:

| Поле | Описание |
|------|----------|
| Риск | Чёткое описание угрозы |
| Митигация | Конкретное действие для снижения риска |
| Ответственный | Функция/человек |
| Дедлайн | Дата решения/принятия решения |

### Шаг 6: SAVE (если --save)

Сохрани результат в `PreMortem-[name]-[YYYY-MM-DD].md` в корне проекта.

## Интеграция с CTX Consilium

```
/ctx-consilium мой PRD --advisors premortem,contrarian
/ctx-consilium готовность к релизу --preset advisor-risk
/ctx-consilium запуск нового продукта --preset advisor-launch
```

## Правила

1. **Честность** — лучше переоценить риск (Tiger), чем недооценить (Paper Tiger)
2. **Конкретность** — "может сломаться" недостаточно. Укажи ЧТО, КОГДА, ПОЧЕМУ
3. **Cross-functional** — рассматривай engineering, design, go-to-market, legal
4. **Не обвиняй** — цель улучшить готовность, а не найти виноватого
5. **Обновляй** — повтори pre-mortem ближе к запуску для проверки митигаций
