# /ctx — Session Workflow System

Система управления контекстом между сессиями. Загружает контекст из GitHub Issues, ведёт логи, сохраняет выводы.

## Использование

- `/ctx` — старт сессии (загрузка контекста + создание лога)
- `/ctx save` — конец сессии (сводка + сохранение в GitHub)

---

## Старт сессии (`/ctx`)

При вызове `/ctx` без аргументов выполни следующие шаги:

### 1. Загрузить контекст из GitHub Issues

```bash
# Открытые задачи
gh issue list -L 20 --json number,title,labels,state

# Недавние уроки (закрытые issues с меткой lesson)
gh issue list -L 10 -l lesson --state closed --json number,title,body

# Текущие блокеры
gh issue list -l blocker --state open --json number,title,body

# WIP задачи
gh issue list -l wip --state open --json number,title,body
```

### 2. Создать файл лога сессии

Создай файл `.sessions/YYYY-MM-DD-HHmm.md` с текущей датой и временем:

```markdown
# Session YYYY-MM-DD HH:mm
**Machine:** [определить из окружения]
**Branch:** [текущая ветка git]
**Goals:** [спросить у пользователя или определить из контекста]

## Context Loaded
- Open issues: [количество]
- Recent lessons: [список]
- Active blockers: [список]

## Actions
<!-- Логировать каждое значимое действие -->

## Errors & Solutions
<!-- При обнаружении и решении бага — документировать здесь -->

## Decisions
<!-- Архитектурные и технические решения -->

## Files Modified
<!-- Список изменённых файлов -->

## Summary
<!-- Заполнить в конце сессии -->
```

### 3. Показать сводку пользователю

Выведи краткую сводку в формате:

```
📋 Сессия начата: YYYY-MM-DD HH:mm
📂 Лог: .sessions/YYYY-MM-DD-HHmm.md

🔓 Открытые задачи: N
  #1 — Название задачи
  #2 — Название задачи

🚫 Блокеры: N
  #X — Описание блокера

📚 Последние уроки:
  #Y — Урок (закрыт)

🔧 WIP:
  #Z — В процессе
```

## В процессе работы

Во время сессии соблюдай следующие правила:

### Логирование действий
После каждого значимого действия добавляй запись в секцию `## Actions` файла сессии:
```
- [HH:mm] действие — файл/компонент — результат
```

### При обнаружении и решении бага
1. Задокументируй в секции `## Errors & Solutions` файла сессии
2. Создай GitHub issue с меткой `lesson`:
```bash
gh issue create --title "Lesson: краткое описание" --label lesson --body "## Ошибка
описание ошибки

## Решение
как исправлено

## Как предотвратить
рекомендации"
```
3. Сразу закрой issue (уроки хранятся как закрытые):
```bash
gh issue close <number>
```

### При принятии архитектурного решения
1. Задокументируй в секции `## Decisions`
2. Если решение значимое — создай issue с меткой `decision`:
```bash
gh issue create --title "Decision: описание" --label decision --body "## Решение
что решили

## Альтернативы
что рассматривали

## Обоснование
почему выбрали это"
```

## Конец сессии (`/ctx save`)

При вызове `/ctx save` выполни:

### 1. Заполнить Summary в файле сессии
Суммируй все действия, ошибки, решения из лога.

### 2. Создать GitHub issue с меткой `session`
```bash
gh issue create --title "Session: YYYY-MM-DD — краткое описание работы" --label session --body "## Сводка сессии
дата, ветка, машина

## Что сделано
- пункт 1
- пункт 2

## Ошибки и решения
- ошибка → решение (ссылка на lesson issue)

## Решения
- решение (ссылка на decision issue)

## Незавершённое
- что осталось доделать

## Изменённые файлы
- список файлов"
```

### 3. Обновить MEMORY.md
Если в ходе сессии обнаружены новые паттерны, уроки, или изменения в архитектуре — обнови соответствующие секции в `MEMORY.md`.

### 4. Обновить WIP issues
- Закрой завершённые WIP issues
- Создай новые WIP issues для незавершённой работы

---

## Важные правила

1. **Логи приватны** — `.sessions/` в `.gitignore`, не попадают в GitHub
2. **Issues публичны** — только финальные выводы, без чувствительных данных
3. **Уроки закрываются сразу** — они хранятся как reference, не как открытые задачи
4. **Session issues закрываются сразу** — это снэпшоты, не задачи
5. **Не дублируй** — если урок уже есть в MEMORY.md, не создавай issue
